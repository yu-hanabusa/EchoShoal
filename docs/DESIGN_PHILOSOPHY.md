# EchoShoal 設計思想

## 1. プロジェクトの根本思想

EchoShoalは「サービスが市場で成功するか」という問いに対し、**多様なステークホルダーの自律的な振る舞いから創発的に答えを導き出す**シミュレーターである。

### 1.1 なぜエージェントベースか

従来の市場予測手法（回帰分析、時系列予測、スコアリング）は「過去のパターンが未来にも当てはまる」という前提に依存する。しかし新規サービスの市場投入は、複数のステークホルダーが互いの行動に反応し合う**複雑適応系**である。

EchoShoalは各ステークホルダーを自律的なエージェントとしてモデリングし、彼らの相互作用から**市場の動態を創発的に生成**する。これにより：

- 「もし大企業が競合サービスを出したら」
- 「もし規制が強化されたら」
- 「もし投資家の資金が枯渇したら」

といったWhat-ifシナリオを、固定的なモデルではなく**エージェントの自律判断として**シミュレーションできる。

### 1.2 核となる設計原則

| 原則 | 説明 |
|------|------|
| **恣意的パラメータの排除** | 重み係数・固定インパクト値・ハードコード数値を使わない |
| **創発的振る舞いの保護** | エージェントの行動制約を最小限にし、予期しない結果を許容する |
| **情報非対称性の再現** | すべてのエージェントが同じ情報を持つわけではない |
| **最小入力・最大出力** | ユーザーはサービス名＋説明文だけで十分。不足はLLMが補間 |
| **検証可能性** | ベンチマークで実在事例との一致度を統計的に検証 |

## 2. 「恣意的パラメータの排除」の原則

### 2.1 問題意識

多くのシミュレーションシステムは以下のようなハードコード値に依存する：

```python
# ❌ やってはいけない例
IMPACT_TABLE = {
    "adopt_service": {"user_adoption": +0.05, "revenue_potential": +0.03},
    "regulate": {"regulatory_risk": +0.08, "user_adoption": -0.02},
}

SUCCESS_SCORE = (
    0.3 * user_adoption
    + 0.2 * revenue_potential
    + 0.15 * tech_maturity
    - 0.1 * competitive_pressure
    ...
)
```

これらの固定値は：
1. **作成者の主観が入る** — なぜ0.05であって0.08ではないのか根拠がない
2. **コンテキストを無視する** — 同じ「規制」でも市場環境により影響は異なる
3. **チューニング地獄に陥る** — パラメータ調整が永遠に続く

### 2.2 EchoShoalのアプローチ

```python
# ✅ EchoShoalの方式
# 市場更新: LLMが文脈を考慮して判断
response = await llm.generate_json(
    system="あなたは市場アナリストです",
    prompt=f"""
    現在の市場状態: {current_dimensions}
    今ラウンドのアクション: {all_actions}
    → 各ディメンションへの影響をJSON形式で出力
    """,
)
# LLMが状況に応じて影響度を判断
# 同じ「adopt_service」でも、市場状況により影響度が変わる
```

**すべての影響度はLLMが文脈を考慮して毎回判断する。** 固定テーブルは存在しない。

### 2.3 安全制約のみ設ける

恣意的パラメータは排除するが、**安全制約**（クランプ値）は設ける：

```
ディメンション delta: -0.1 〜 +0.1 にクランプ
マクロ指標 delta:    -0.05 〜 +0.05 にクランプ
全値:               0.0 〜 1.0 にクランプ
```

これは「1ラウンドで市場が激変することは現実的でない」という物理的制約であり、恣意的な重みではない。

## 3. 創発的振る舞いの設計

### 3.1 エージェント自律性

各エージェントは**LLMが性格パラメータに基づいて自律的に行動を選択**する。開発者が「企業Aはこう動くべき」と決め打ちしない。

```
┌────────────────────────┐
│   エージェントの意思決定   │
│                         │
│  入力:                   │
│  ├─ 市場状態 (8次元)      │
│  ├─ 可視な他者の行動       │
│  ├─ 自身の過去の行動       │
│  ├─ 性格パラメータ        │
│  └─ 文書から抽出した知識   │
│                         │
│  プロセス:               │
│  ├─ LLMが状況を分析       │
│  ├─ 性格バイアスを適用     │
│  └─ JSON形式でアクション選択 │
│                         │
│  出力:                   │
│  ├─ action_type          │
│  ├─ description (理由)   │
│  ├─ impact (市場への影響) │
│  └─ self_impact (自己影響) │
└────────────────────────┘
```

### 3.2 性格パラメータ（認知バイアス）

性格は「正解の行動」を決めるものではなく、**判断の偏りを生むもの**として設計されている。

| パラメータ | 効果 | 創発への影響 |
|-----------|------|------------|
| conservatism (保守性) | 高い→現状維持を好む | 市場の慣性を生む |
| bandwagon (同調性) | 高い→他者を真似る | トレンドの加速/バブルを生む |
| overconfidence (過信) | 高い→リスクを過小評価 | 過剰投資→失敗パターンを生む |
| sunk_cost_bias | 高い→過去の投資を捨てられない | 撤退の遅延を生む |
| info_sensitivity | 低い→情報を見落とす | 情報格差を生む |
| noise (ノイズ) | LLM判断後にランダム行動に差替 | 予測不能性を確保 |

### 3.3 ノイズの役割

`noise`パラメータ（0〜0.3）は、LLMの判断後にランダムなアクションに差し替える確率を制御する。これは：

- LLMが常に「合理的」な判断をすることへの対抗措置
- 現実世界の非合理的な行動（突然の方針転換、感情的判断）の再現
- シミュレーション結果の多様性確保

## 4. 情報非対称性の設計

### 4.1 可視性制御

現実の市場では、すべてのプレイヤーが同じ情報を持つわけではない。EchoShoalはこれを**3段階の可視性**で再現する。

```
┌──────────────────────────────────────────────────┐
│                 可視性制御モデル                     │
│                                                    │
│  public  ─── 全エージェントに見える                  │
│              例: サービス採用、競合リリース、規制導入    │
│                                                    │
│  partial ─── 同種ステークホルダーに見える              │
│              例: R&D投資 → 企業間のみ共有             │
│              例: divest → 投資家間のみ共有            │
│                                                    │
│  private ─── 本人のみ見える                          │
│              例: 不採用判断、プロジェクト放棄、様子見    │
└──────────────────────────────────────────────────┘
```

### 4.2 GraphRAGによるコンテキスト構築

各エージェントが意思決定する際、GraphRAGRetrieverが**そのエージェントから見える情報だけ**を収集する：

```
エージェントAのコンテキスト:
├── 自分の過去の行動（全て見える）
├── 他エージェントのpublicアクション（全て見える）
├── 同業他者のpartialアクション（条件付きで見える）
├── 文書から抽出されたエンティティ情報
├── 市場スキル需要の集計
├── アクティブなイベント情報
└── 統計データ（e-Stat）

※ 他エージェントのprivateアクションは見えない
```

## 5. LLM情報補間の設計

### 5.1 最小入力の原則

ユーザーが入力するのは最小限：

```
必須: シナリオ説明文（10〜2000文字）
任意: サービス名、GitHub URL、シミュレーション期間、文書
```

### 5.2 LLMが補間する情報

不足情報をLLMが類似サービスの知識に基づいて推定する：

```
ユーザー入力                     LLM補間
「チーム向けチャットツール」  →  収益モデル: SaaS月額制
                              推定競合: Slack, Teams, Discord
                              ターゲット: 中小企業のリモートチーム
                              技術スタック: React + Node.js
                              チーム規模: 5-10人スタートアップ
                              市場規模: 国内500億円規模
                              初期市場状態: 各ディメンション0.0〜1.0
```

### 5.3 補間の透明性

補間された情報は`confidence_notes`として記録され、レポートで「推定値」として明示される。ユーザーが追加文書を提供すれば、推定精度が向上する。

## 6. OASISエンジンの採用理由

OASISフレームワーク（CAMEL-AI）を採用し、SNS空間でのエージェント間インタラクションを実現している。

| 特性 | 説明 |
|------|------|
| インタラクション | SNS投稿型（投稿・コメント・リポスト・フォロー） |
| 関係グラフ | フォロー・コメントから自動成長 |
| エージェントの「考え」 | 投稿内容として可視化される |
| スケーラビリティ | 最大200体 |
| プラットフォーム | Reddit（ビジネス議論に適した長文形式） |
| 外部依存 | Ollama + CAMEL-AI + SQLite + Neo4j |

## 7. ベンチマーク評価の設計

### 7.1 検証可能性の確保

シミュレーターの出力が「妥当かどうか」を客観的に検証するため、**実在のサービスリリース事例**をベンチマークとして使用する。

```
ベンチマーク定義:
├── シナリオ: 「2014年、チーム向けビジネスチャットツールを投入」（Slack）
├── 期待トレンド:
│   ├── user_adoption: UP     ← 実際にSlackは急成長した
│   ├── competitive_pressure: UP ← 競合が多数参入した
│   └── funding_climate: UP   ← SaaS投資が活況だった
└── 参照URL: Wikipedia等の公開情報
```

### 7.2 恣意的スコアリングの排除

旧方式（廃止済み）：
```python
# ❌ 旧方式: 恣意的な重み
score = 0.6 * direction_accuracy + 0.3 * magnitude + 0.1 * correlation
```

現方式：
```
期待: user_adoption → UP
実際: user_adoption が 0.3 → 0.7 に上昇 → UP → 方向一致 ✅

スコア = 方向が一致したトレンド数 / 全トレンド数
```

**単純な方向一致率のみ**をスコアとする。「どれくらい上がるべきか」の数値は恣意的になるため使用しない。

### 7.3 統計的評価

LLMの確率的性質により、1回の実行結果は再現性が低い。そのため**複数回実行**して統計的に評価する：

```
ベンチマーク: Slack 2014
実行回数: 5回
結果:
  Run 1: 方向一致率 80% (4/5)
  Run 2: 方向一致率 60% (3/5)
  Run 3: 方向一致率 80% (4/5)
  Run 4: 方向一致率 100% (5/5)
  Run 5: 方向一致率 60% (3/5)

平均方向一致率: 76%
標準偏差: 15.2%
→ シミュレーターは概ね妥当な方向性を予測できている
```

## 8. セキュリティ設計原則

| 原則 | 実装 |
|------|------|
| 秘密情報のゼロハードコード | 全APIキー・パスワードをpydantic-settings経由で環境変数から取得 |
| 入力境界でのバリデーション | Pydanticモデルで全APIパラメータを検証 |
| LLMプロンプトインジェクション防止 | ユーザーテキストをサニタイズしてからプロンプトに注入 |
| Cypherインジェクション防止 | Neo4j全クエリをパラメータ化（`$variable`構文） |
| CORS制限 | localhost:5173/5174/5175のみ許可 |
| レート制限 | 同時実行最大3、毎分最大10リクエスト |

## 9. 技術選定の根拠

| 技術 | 選定理由 |
|------|---------|
| **FastAPI** | async対応でLLM呼び出しの非同期処理に適切。Pydanticとの統合が強力 |
| **Neo4j** | エージェント間関係・因果チェーンの表現にグラフDBが最適。RDBのJOINでは表現困難 |
| **Redis** | ジョブ状態管理・キャッシュに軽量高速。永続化不要（7日TTL） |
| **Ollama** | ローカルLLMでプライバシー保護。APIコスト削減。低レイテンシ |
| **Claude API** | レポート生成など高品質な文章生成が必要なタスクに使用 |
| **ルールベース辞書NLP** | 技術名・政策名は正規表現辞書で高速・確定的に抽出。組織名はLLMが担当 |
| **React + TypeScript** | 型安全性とコンポーネント再利用性。D3/Rechartsとの統合が容易 |
| **React Query** | サーバー状態管理の標準。ポーリング・キャッシュ・楽観的更新をカバー |
| **Tailwind CSS** | ユーティリティファーストでコンポーネントごとのスタイリングが効率的 |
| **OASIS (CAMEL-AI)** | 大規模エージェントシミュレーションの研究フレームワーク。SNS空間の再現に特化 |
